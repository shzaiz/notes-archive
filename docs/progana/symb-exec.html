注记. (1) 为什么要使用: 为了解决路径爆炸.

详细解释

在符号执行中，每遇到一个条件分支（如 if (e)），都会将执行路径分成两个

  • 条件成立：路径约束变为 pc ∧ e

  • 条件不成立：路径约束变为 pc ∧ ¬e

这会导致路径数随着分支数呈指数级增长——即路径爆炸。

(2) Merge部分是做什么的? 还是合并等价状态. 减少路径爆炸

详细介绍

当我们在执行完一条指令后产生多个 successor 状态（S），我们想要把“等价”的状态合并，从而减少后续要探索的路径数量。

这个过程叫做 状态合并（state merging），目的是控制状态爆炸问题。

forall (l”, pc', s') ∈ S : 遍历所有新生成的 successor 状态。

if : 查找工作集中（worklist w）是否已经有一个“相似”的状态。其中：

  • (l”, pc”, s”) 是旧状态（在 中）

  • (l”, pc', s') 是新状态（在 中）

  • ∼ 是一个相似性关系（similarity relation），用于判断两个状态是否可以合并。

一般来说，这里的相似性关系通常要求：

  • 程序位置 l” 相同；

  • 状态结构（变量集合等）相似；

  • 值域兼容（即变量有相同含义）。

那就先从 worklist 中移除旧的等价状态。添加一个 合并后的新状态：

  • 程序位置仍然是 ℓ′′

  • 合并 path condition：pc′ ∨ pc′′

  • 合并的符号状态 s[v] 是一个条件表达式：

其中s[v] = ite(pc', s'[v], s”[v])的意思是如果 pc' 成立，就用 s'[v]，否则用 s”[v]。这其实是在构造一个“分支敏感”的符号状态。如果没找到“相似”的旧状态，就直接把新状态加入。
